# Robot Sync App (Clean Architecture)

This is a clean-architecture orchestrator for:
- Speech (Riva)
- Finger gestures (Arduino)
- Face expressions (LCD adapter)
- Future wheels/navigation (ROS2 + Unitree LiDAR)
- Future user perception (Kinect)

Main arm servos are safety-disabled by default.

## Safety

In [config/config.yaml](config/config.yaml):
- `safety.enable_main_arms: false`

This blocks arm-related gesture commands in the gesture adapter.

## Structure

- `src/robot_sync_app/domain`: core models/events
- `src/robot_sync_app/ports`: interfaces
- `src/robot_sync_app/application`: orchestration logic
- `src/robot_sync_app/adapters`: infrastructure implementations
- `src/robot_sync_app/bootstrap`: dependency wiring
- `config/config.yaml`: environment config

## Run

From [robot_sync_app](.):

```bash
pip3 install -r requirements.txt
```

### Text mode (manual text -> speech/gesture/face)

```bash
PYTHONPATH=src python3 -m robot_sync_app.main --config config/config.yaml --text "Hello Adrian, welcome to quiz time!" --intent quiz
```

### Voice mode (mic -> Riva ASR -> Bedrock Llama -> Riva TTS)

```bash
PYTHONPATH=src python3 -m robot_sync_app.main --config config/config.yaml --voice --intent chat
```

Optional bounded session:

```bash
PYTHONPATH=src python3 -m robot_sync_app.main --config config/config.yaml --voice --intent quiz --max-turns 5
```

## Notes

- `ArduinoSerialGestureAdapter` sends compact serial lines; for `fingers_wave` it uses `TALK_ON` / `TALK_OFF` for robust live decoding.
- Voice mode uses microphone input (PyAudio) and Riva ASR.
- LLM replies are generated by AWS Bedrock (`llm.aws_model_id`, e.g. Llama 70B).
- Replace `LCDStubFaceAdapter` with your real Jetson LCD drawing adapter.
- Keep gesture names to allowed finger presets from config.
- Wheel/autonomy modules are scaffolded but disabled by default.

## Gesture serial root cause and fix (March 2026)

### Root cause
- During live chat, serial framing/decoding was fragile: some command lines were dropped before decode on Arduino.
- Arduino logs showed buffer reset events and missing `Matched ...` / `ACK TALK_ON` lines, so talk motion state never activated.

### Fix
- Added a deterministic wire mapping in [`src/robot_sync_app/adapters/gesture/arduino_serial.py`](src/robot_sync_app/adapters/gesture/arduino_serial.py):
	- `fingers_wave` start -> `TALK_ON`
	- `fingers_wave` stop -> `TALK_OFF`
- Hardened serial send path:
	- clear stale input before send,
	- flush after write,
	- retry once if no response is observed.
- Added receive/decode instrumentation to Arduino debug sketch [`../Humanoid/arduino/finger_servos/talk_motion_serial.ino`](../Humanoid/arduino/finger_servos/talk_motion_serial.ino).

### Validation tools
- Diagnostic script to compare exact sent bytes vs Arduino decode logs:
	- [`../Humanoid/examples/test_scripts/test_talk_motion_serial.py`](../Humanoid/examples/test_scripts/test_talk_motion_serial.py)
